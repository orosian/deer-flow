## TL;DR

v1 closes the **read-side** of the integration: the backend exposes
the preset catalog, the frontend renders a PresetSelector dropdown,
and `hooks.ts` already translates `context.preset_id` into
`assistantId = "gh:<id>"` on the wire.

v1 **does not** close the **write-side**: the sidebar "New chat"
button does not yet prompt for a preset, and the chosen preset is
not yet persisted into LangGraph thread metadata. v1.1 owns that.

The component is hidden-disabled by default (`aria-disabled`),
which means a thread started without a preset works exactly as before
— `context.preset_id` is undefined → `assistantId = "lead_agent"`.

## Why Each Change Was Made

<Steps>

### Backend router — why a separate file instead of stuffing it into `models.py`

The preset catalog is a different domain (graph-harness execution
paradigm) from models (LLM choice). A separate router:

* Keeps the SEC-1 whitelist audit surface obvious — anyone reading
  `routers/graph_presets.py` sees that the whitelist is consulted
  before any preset is returned to the frontend.
* Lets us register/unregister the entire preset path with one line
  flip in `app.py` if graph-harness is intentionally disabled.
* Makes the router independently testable without standing up the
  whole gateway — the tests patch `_discover_presets` /
  `_load_allowed_presets` directly.

### Why `applyThreadPresetOverride` mirrors `applyThreadModelOverride`

Because preset selection follows the **exact same lifecycle** as
model selection:

* User picks at thread creation time
* Per-thread override (different threads can use different presets)
* Survives page reload (localStorage)
* Re-applied at the top of `useThreadSettings` via
  `applyThreadModelOverride` → `applyThreadPresetOverride` chain

Mirroring means **no new mental model** for future maintainers — the
preset code path is structurally identical to the model code path.

### Why `useActivePreset` lives in `preset-selector.tsx`, not `graph-presets/hooks.ts`

`useGraphPresets` returns the raw list. `useActivePreset` returns
*the one preset that matches a given id* (or a "Default Lead Agent"
fallback). The lookup is UI-specific (decides what to render in the
trigger chip), so it lives next to the component, not next to the
data fetcher. **Rule of thumb**: data fetching → `core/`, view
transformation → `components/`.

### Why `aria-disabled` instead of HTML `disabled` on the trigger

HTML `disabled` on a `<button>` swallows **all** pointer events,
including the hover that drives the locked-tooltip. We need the
tooltip to be visible to explain **why** the user cannot change the
preset. `aria-disabled` keeps the visual disabled look + cursor,
preserves hover, and the `onClick preventDefault` stops navigation.
This pattern is reused from elsewhere in the codebase — search
`aria-disabled="true"` to find prior art.

### Why `Translations.presetSelector?` is optional

We did not want to block v1 on a Chinese translation. Marking the
field optional lets `zh-CN.ts` stay untouched. Consumers must use
`t.presetSelector?.label ?? "Preset"` defensively until the zh-CN
stub is added in v1.1.

### Why `useThreadSettings` is composed instead of given a new hook

`useThreadSettings(threadId)` already exists and is called by many
downstream consumers (`InputBox`, `WorkspaceHeader`, …). Composing
`applyThreadPresetOverride` inside it means **every existing caller
automatically gets `preset_id` on `settings.context` without
changing their call sites**. If we had shipped a new
`useThreadPreset(threadId)`, every consumer would have had to call
both hooks and merge results.

</Steps>

## Thread-id Stability Problem (read this before v1.1)

The settings store keys everything on `threadId`. For an **existing**
thread, `threadId` is the backend-issued UUID — stable across page
reloads. For a **new** thread (`/workspace/chats/new`), `threadId`
is a **client-side pre-allocated UUID** (see
`src/components/workspace/chats/use-thread-chat.ts:33-39`) that gets
**replaced** by the backend-issued UUID on `onStart` (see
`src/app/workspace/chats/[thread_id]/page.tsx:101`).

**Consequence**: if a user clicks "New chat" → picks a preset (stored
under key `deerflow.thread-preset.<temp-uuid>`) → sends first message →
backend creates thread with new id → `useThreadSettings` reads from
`deerflow.thread-preset.<backend-uuid>` → **preset is lost**.

Three ways to fix this; v1.1 must pick one:

1. **URL-lock pattern**: store the preset in the URL (`?preset=echo`),
   not in settings. The page reads it from `searchParams` on every
   render. Cheap, but breaks the model-style "localStorage for
   per-thread overrides" pattern.
2. **Two-stage settings**: introduce a `pendingPresetId` global that
   `hooks.ts`'s `onCreated` callback copies into the thread-local
   storage using the *new* threadId at thread creation time.
   Conceptually clean, but requires a new global setting type.
3. **Backend-eager pattern**: change the backend adapter to accept
   `metadata.preset_id` at thread creation (not just at run time),
   and have the frontend pass it on `client.threads.create()`. Most
   correct (preset becomes part of thread metadata), but requires
   backend work.

**Recommendation**: option 2 (two-stage settings). It is a frontend-only
change, fits the existing settings store pattern, and the
backend-eager pattern can be added later as an optimisation.

## Subagent Process Notes (for the curious)

This commit was built by three parallel Claude Code subagents
(A, B, C) plus direct Hermes edits. Useful to know:

* **A** (types + data + store + i18n + **also touched `hooks.ts:528`**)
  exceeded its scope to fix the assistantId wiring because B's prompt
  depended on it. Net win — fewer files for B to touch.
* **B** (PresetSelector component + input-box wiring) ran twice.
  First time it correctly braked waiting for A. Second time it
  succeeded but only got as far as imports + state; Hermes finished
  the JSX and removed a duplicate import.
* **C** (sidebar + thread metadata) failed twice with
  `Error: Reached max turns`. Root cause: prompt asked for 4 things
  at once (pitfall from subagent-recovery R6). v1.1 prompt should
  be **one thing per subagent**.
* `useThreadSettings` was not composed by A — Hermes added it
  because B's prompt assumed it existed.

Lessons for v1.1: **single-responsibility prompts**, pre-flight
file existence check before "modify line N", and always have
Hermes run a final smoke test (pytest + grep on changed files)
before declaring done.
