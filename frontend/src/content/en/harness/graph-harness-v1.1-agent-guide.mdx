## 3. The Thread-id Stability Problem (mandatory reading)

The `settings` store keys everything on `threadId`. From
`use-thread-chat.ts:33-46`:

```
newThreadIdRef.current = threadIdFromPath === "new" ? uuid() : null;
```

For `/workspace/chats/new`, `threadId` is a **client-side
pre-allocated UUID**. On first message send, `onStart` calls
`setThreadId(backendThreadId)` which **changes** the value. Any
per-thread setting stored under the temp UUID is **lost** because
`useThreadSettings` reads from
`localStorage["deerflow.thread-preset." + threadId]`.

**Pick one of the three options** (in order of recommendation):

### Option A (recommended): URL-lock pattern

Store the preset in the URL query string, not in settings.

* Sidebar popover navigates to
  `/workspace/chats/new?preset=echo` (or to a stub
  `/workspace/chats/new` with a `?preset=` query).
* `page.tsx` reads `searchParams.get("preset")` on every render and
  passes it into the `useThreadStream` `context` argument via the
  existing `applyThreadPresetOverride`-equivalent (you will need a
  new local hook — see `useUrlPreset` sketch below).
* Persistence into thread metadata is a **separate write** in
  `onCreated` (the existing `agent_name` pattern at `hooks.ts:561-567`).

```ts
// src/core/graph-presets/use-url-preset.ts (sketch)
export function useUrlPreset(): string | undefined {
  const searchParams = useSearchParams();
  return searchParams.get("preset") ?? undefined;
}
```

**Pros**: simplest, no settings-store changes, no UUID churn. The
URL is the source of truth — refresh preserves it.

**Cons**: long URLs if the preset id is long (it isn't — current
ids are `echo/echo`, `coding/coding_pipeline`).

### Option B: Two-stage settings

Add a `pendingPresetId` field to global `LocalSettings.context`.
On `onCreated`, copy it into the backend thread's metadata AND into
the now-stable `useThreadSettings(backendThreadId)`.

**Pros**: matches existing per-thread-settings pattern. URL stays
clean.

**Cons**: requires a new global setting type + a migration when
`pendingPresetId` is consumed. Two writes per thread creation.

### Option C: Backend-eager thread creation

Change `client.threads.create(...)` to accept `metadata: { preset_id }`
at creation time. Backend adapter reads `metadata.preset_id` in
`make_graph_harness_agent` and uses it for routing (instead of
deriving from `assistant_id`).

**Pros**: most correct. Preset is part of the thread from t=0.

**Cons**: requires backend work. Must coordinate with
`backend/app/gateway/services.py:444-465` where thread creation is
already plumbed. Out of scope for "frontend-only v1.1".

**Pick A** unless you have a strong reason not to.

## 5. Failure Recovery

### Symptom: `useStream` reconnects in a loop after the page reloads

**Cause**: you put `preset_id` into the `useStream` `context` argument,
which is part of the `useStream` config. When the context object
identity changes (URL navigation, settings re-render), `useStream`
sees a new config and reconnects.

**Fix**: derive `preset_id` **inside** the `onCreated` callback, not
in the `useStream` config. Only the wire-level `assistantId` should
change; `context.preset_id` is consumed at message-send time, not at
stream-init time.

### Symptom: backend logs show `make_lead_agent` instead of `make_graph_harness_agent`

**Cause 1**: `context.preset_id` is undefined at `useStream` time
because the `useUrlPreset` hook is not yet read by `page.tsx`.
Fix: ensure `page.tsx` reads `useUrlPreset()` and forwards it.

**Cause 2**: `metadata.preset_id` is being read by the backend but
`assistant_id` is not "gh:\{preset\}". Fix: `hooks.ts:528` already
maps `context.preset_id → "gh:\{id\}"`. Verify `context.preset_id` is
populated when the message is sent (add a temporary `console.log`).

### Symptom: Page reload loses the preset

You picked Option B or C but the URL doesn't carry `?preset=`.
Either switch to Option A, or migrate the preset into URL at
`onStart` time via `history.replaceState`.

### Symptom: Subagent (Claude Code) keeps hitting `Reached max turns`

**Cause**: prompt is too long / too many tasks. Subagent prompt must
fit in ≤ 50 lines for this work. Cut each Step above into its own
subagent run.
