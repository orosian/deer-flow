# Graph Harness Integration (Frontend)

<Callout type="info" emoji="🧩">
  This document describes every frontend change that participates in the
  graph-harness integration. Backend routing and security policy live in
  `deer-flow/backend/app/gateway/adapters/graph_harness.py` — the frontend
  only mirrors the contract the backend already implements.
</Callout>

## Background

The DeerFlow backend exposes **two sibling execution paradigms**, selected by
the `assistant_id` shape sent on every LangGraph run:

| `assistant_id` value          | Backend factory              | Paradigm                              |
| ----------------------------- | ---------------------------- | ------------------------------------- |
| `"lead_agent"` (default)      | `make_lead_agent`            | LLM-driven super agent + subagents    |
| `"gh:<preset>"` (e.g. `gh:echo`) | `make_graph_harness_agent`   | Compiled preset graph (deterministic) |

Both factories return a thin proxy that feeds the same `RunManager` +
`MemoryStreamBridge` + `sse_consumer` plumbing. The SSE wire format is
byte-identical between the two paths, so the frontend does not need a separate
stream decoder — only a different `assistantId` parameter on the LangGraph SDK
call.

The frontend's job in this integration is therefore narrow:

1. **Let the user pick a preset** before starting a new thread.
2. **Translate the selection** into `assistantId = "gh:<preset>"` on every run
   inside that thread.
3. **Lock the preset per thread** — it cannot be changed mid-thread (see
   "Per-thread locking" below).

## Per-thread locking (Design Decision)

<Callout type="warning" emoji="🔒">
  Once a thread has been created with a preset, **the preset cannot be
  changed** for that thread. Switching presets opens a new thread.
</Callout>

**Why.** A `gh:<preset>` run executes a compiled, deterministic graph that
loads preset manifests, validates port contracts, and expects a specific
blackboard schema. Re-targeting an in-flight thread to a different preset (or
back to `lead_agent`) would mix two execution paradigms inside the same
LangGraph checkpoint history and is not supported by the backend adapter. The
same constraint already exists for the `agent_name` dimension — switching the
custom agent on an existing thread is not supported either.

**Implementation.** `preset_id` is stored in `AgentThreadContext` and mirrored
to LangGraph thread `metadata.preset_id` on creation (mirroring the existing
`agent_name` pattern at `src/core/threads/hooks.ts:539`). Reads of the context
prefer the thread metadata when present, falling back to the local default.
Writes that would change the preset on an existing thread are rejected with
a toast.

## Data Contracts

### `AgentThreadContext` (extended)

```ts
export interface AgentThreadContext extends Record<string, unknown> {
  thread_id: string;
  model_name: string | undefined;
  thinking_enabled: boolean;
  is_plan_mode: boolean;
  subagent_enabled: boolean;
  reasoning_effort?: "minimal" | "low" | "medium" | "high";
  agent_name?: string;
  preset_id?: string;   // NEW — null/absent means default lead_agent path
}
```

### Backend response: `GET /api/graph-presets`

```ts
export interface GraphPreset {
  id: string;             // e.g. "echo" — combined with "gh:" by the frontend
  display_name: string;   // e.g. "Echo"
  description: string;
  category: string;       // e.g. "utility"
}

export interface GraphPresetsResponse {
  presets: GraphPreset[];
}
```

### LangGraph thread metadata

The `preset_id` is mirrored to LangGraph `Thread.metadata.preset_id` so it
survives page reloads and is visible to the backend adapter. This mirrors the
existing `agent_name` field (see `hooks.ts:539-567`).

## Routing Logic in `hooks.ts`

```ts
// src/core/threads/hooks.ts — line 528
const thread = useStream<AgentThreadState>({
  client: getAPIClient(isMock),
  assistantId: context.preset_id ? `gh:${context.preset_id}` : "lead_agent",
  // … unchanged
});
```

The `context` object is read from `useThreadSettings(threadId)` (which now
honours a per-thread `preset_id` override). The mapping is intentionally
pure: empty / undefined `preset_id` keeps the historical `lead_agent`
default, so any thread created before this change continues to work.

## `PresetSelector` UI Behaviour

<Steps>

### New thread (no thread id yet)

The sidebar "New chat" button opens a small popover with a list of presets
sourced from `useGraphPresets()`. The first item is **"Default Lead Agent"**
(id = `undefined`), which preserves today's behaviour. Selecting a preset
navigates to `/workspace/chats/<new-uuid>` with the preset pre-selected and
the selector displayed as a read-only chip ("Preset: gh:echo").

### Existing thread

`<PresetSelector>` is rendered next to `<ModelSelector>` but is **disabled**
and shows the locked preset id (or "Lead Agent"). Hovering the chip shows a
tooltip: *"Preset is locked to this thread. Create a new thread to use a
different preset."*

### Switching back to lead agent

There is no "switch" path. The user must click "New chat" → choose "Default
Lead Agent" — which creates a brand-new thread with `preset_id = undefined`.

</Steps>

## Error Handling

The backend adapter distinguishes three error codes:

| Code | Meaning                                | Frontend behaviour                              |
| ---- | -------------------------------------- | ----------------------------------------------- |
| 400  | Pattern violation on preset name       | (cannot happen — preset is selected from a list, not typed) |
| 403  | Preset not in whitelist                | Hide the preset from the dropdown (filter on response)        |
| 404  | Preset not found                       | Hide the preset from the dropdown (filter on response)        |

The frontend never sends an arbitrary preset name to the backend — it only
sends ids it received from `GET /api/graph-presets`. The 403/404 paths are
defence-in-depth only.
